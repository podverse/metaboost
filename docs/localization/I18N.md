# Internationalization (i18n)

The boilerplate uses a three-tier i18n flow: **originals** → **overrides** → **compiled**. Runtime messages are loaded from **compiled**; the compiled directory is generated at build time and is not committed.

## Layout (per app)

- **`i18n/originals/`** — Source of truth. `en-US.json` is maintained by hand; other locales can be added (e.g. LLM-generated). Committed.
- **`i18n/overrides/`** — Human corrections for non–en-US locales. Same key structure as originals; non-empty override values win over originals. Committed.
- **`i18n/compiled/`** — Merged output (originals + overrides). Generated only; **not** committed (in `.gitignore`).

## Commands

- **`npm run i18n:sync`** (root) — Syncs originals from en-US for web, management-web, and helpers-i18n: adds any keys present in en-US but missing in other locales’ originals with **empty string** (do not put English in es or other locale files; those must be in the target language, e.g. Spanish for es). Run after adding keys to en-US; then translate empty keys to the locale language.
- **`npm run i18n:compile`** (root) — Compiles i18n for `apps/web`, `apps/management-web`, and `packages/helpers-i18n`. Syncs override structure from each locale’s originals, then merges originals + overrides into compiled. Each app also has its own `i18n:compile` script; `prebuild` runs it before `next build`.
- **`npm run i18n:validate`** (root) — Validates that all locale files have the same keys and key order as en-US; en-US originals must have no empty values; non–en-US originals may have empty (pending translation). Overrides match structure. Validates web, management-web, and helpers-i18n. Run in CI.

## Sync and workflow (Podverse-style)

When en-US changes are pushed to **develop**, `.github/workflows/i18n.yml` runs: **sync** originals from en-US (missing keys get empty string; do not put English in es or other locale files), **compile**, **validate**, then commits and pushes originals + overrides if changed (`chore: sync i18n keys from en-US [skip ci]`). New empty keys should then be translated to the target language (e.g. Spanish for es). Locally, run `npm run i18n:sync` then `npm run i18n:compile` after adding keys to en-US.

## Locale language (non–en-US)

Non–en-US originals (e.g. `es.json`) must be in the **target language** (e.g. Spanish for es). Do not put English text into Spanish or other locale files. The sync script adds missing keys with empty string; fill them with the correct translation.

## Adding or changing keys

1. Edit **`i18n/originals/en-US.json`** in the app (e.g. `apps/web` or `apps/management-web`). Use nested keys and namespaces (e.g. `auth.login.title`, `common.loading`, `ui.auth.login.email`). Do not leave empty values in en-US.
2. Run **`npm run i18n:sync`** then **`npm run i18n:compile`** (or merge to develop and let the i18n workflow do it). Translate any new empty keys in non–en-US originals to the locale language. Run **`npm run i18n:validate`** to confirm.
3. Use the key in code: `useTranslations('auth')` then `t('login.title')`, or `useTranslations('ui')` then `t('auth.login.email')` for UI package keys.

## Key conventions

- **Namespaces:** `auth`, `common`, `dashboard`, `settings`, `errors`, **`ui`** (see below).
- **UI package:** Components in `@boilerplate/ui` that render their own text use the **`ui`** namespace. Each app must include a full **`ui`** object in its `i18n/originals/en-US.json` (and other locales). Required keys are listed in [packages/ui/I18N-KEYS.md](../../packages/ui/I18N-KEYS.md).

## Locale detection

- Cookie `NEXT_LOCALE` (if supported) → Accept-Language header → default locale (see below).
- **Default locale:** Set `DEFAULT_LOCALE` (e.g. `en-US`). If unset, defaults to `en-US`.
- **Supported locales:** The code has a hardcoded list of all locales that can have originals/compiled (currently `en-US` and `es`). Set `SUPPORTED_LOCALES` to restrict which locales are active:
  - Unset or `all-available`: use the full hardcoded list.
  - Comma-delimited list (e.g. `en-US,es`): only those locales appear in the app; values must be in the hardcoded list.
- **Sync points when adding a locale or changing default/supported:** Update **`packages/helpers/src/locale/constants.ts`** (`ALL_AVAILABLE_LOCALES`, `DEFAULT_LOCALE`). This is the single source of truth; web and management-web import from `@boilerplate/helpers`; backend (helpers-i18n) re-exports from helpers. Document `DEFAULT_LOCALE` and `SUPPORTED_LOCALES` in the sidecar env templates for web and
  management-web (`apps/web/sidecar/.env.example`,
  `apps/management-web/sidecar/.env.example`); the app `.env.example` files contain only
  `RUNTIME_CONFIG_URL`.

## Backend i18n (API and management-api)

User-facing messages from the APIs (password validation errors, email subject/body) use **`@boilerplate/helpers-i18n`**, which follows the same originals → overrides → compiled pattern. Locale is resolved from the **Accept-Language** header via `resolveLocale()`; default and supported locales match the web apps (`en-US`, `es`).

- **Where it’s used:** Password validation messages (returned in 400 responses) in both **api** and **management-api**; email subject/text/html for verification, password reset, and email-change in **api** only (management-api does not send transactional email).
- **Package:** `packages/helpers-i18n` — `i18n/originals/`, `i18n/overrides/`, `i18n/compiled/` (compiled is generated, not committed). Namespaces: `password.*`, `email.*`.
- **Adding/changing keys:** Edit `packages/helpers-i18n/i18n/originals/en-US.json` (and other locales for key parity), run `npm run i18n:compile` and `npm run i18n:validate`. Use `t(locale, 'password.required')` or `getPasswordValidationMessages(locale)` / `getVerificationEmailContent(locale, link)` etc. in API code.
- **Not translated:** Internal or developer-facing error messages (e.g. "Server configuration error", "Authentication required") are left in English and are not passed through i18n.

## Storybook

Storybook is out of scope for i18n. No translation provider or message bundle is required for stories; they keep using current hardcoded or default text.

## Generating translations

Translations for non–en-US locales can be generated with an **LLM of your choosing** (e.g. Cursor chat, another AI assistant, or manual translation). Keep the same key structure as `en-US.json` in each locale file; then run `npm run i18n:compile` and `npm run i18n:validate` to sync overrides and confirm key parity.
